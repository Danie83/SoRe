Introduction

SoRe (Social Recommender) is a Web modular system able to recommend certain items of interest (people, events, places, other things) according to a given knowledge graph expressed in RDF and automatically built for a specific user - for example, based on her/his social media profile(s) - by considering multiple similar features/properties. For instance, suggesting the members of an IT team based on desired skills (excellent knowledge of certain areas like Web technologies + open hardware), geographic location (i.e. from Romania and Chile only), technical preferences (e.g., using free software), background info (demographics, education, occupation history, driving license, other competencies), hobbies (i.e. horror movies + classical music), aversions (e.g., communication by phone, sport, politics) and so on. The system should be “smart” enough to improve recommendations based on various methods such as user feedback, reasoning, and/or machine learning. The recommended items will be available via a SPARQL endpoint.

Requirements

These following requirements form a foundation for developing and evaluating the web modular system. Each requirement should be testable and measurable to ensure that the system meets the specified criteria.

Functional Requirements

User Profile Creation

The system should allow users to create and update their profiles. This involves user registration by either creating an account or logging using their Google account, in this case some of the information can be acquired beforehand.

Knowledge Graph Construction

The system must automatically build a knowledge graph for each user based on the imported data, representing relationships between the user and various items of interest.

Recommendation Generation

the system should generate personalized recommendations for users based on their knowledge graphs, considering features such as skills, geographic location, technical preferences, background information, hobbies , and aversions.

User Feedback Mechanism

Users should be able to provide feedback on the recommendations received, influencing the system's future recommendations.

SPARQL Endpoint Access

The system must expose a SPARQL endpoint through which users can query and retrieve their personalized recommendations.

Continuous Improvement

The system should incorporate machine learning algorithms to continuously improve recommendation accuracy based on user interactions and feedback.

Non-functional Requirements

Performance

The system should respond to user queries within a reasonable time frame (e.g. under 3 seconds) to ensure a responsive user experience.

Scalability

The system should be able to handle a growing number of users and an expanding knowledge graph without a significant degradation in performance.

Security

User data, especially sensitive information obtained from social media profiles, must be securely stored and transmitted. The system should implement encryption and access controls.

Reliability

The system should be available for use 99.9% of the time, minimizing downtime for maintenance or unexpected issues.

User privacy

The system must adhere to privacy regulations and ensure that the user data is handled responsibly. Users should have control over the information shared and used for recommendations.

Adaptability

The system should be compatible with a variety of devices and browsers to ensure a seamless user experience.

Maintability

The system should be designed with a modular and maintainable code to facilitate updates, bug fixes, and future enhancements.

Documentation

Comprehensive documentation should be provided for developers, system administrators, and end-users to understand the system’s functionality, configuration, and usage.

System Architecture

Django Application

Responsibility: Handles user interactions through a web-based user interface.

Components:

  • User Interface: Provides forms and views for user registration, profile management, and feedback submission.

  • User Authentification: Manages user login, registration, and profile updates usings a dedicated database for user information.

Flask Application

Responsibility: Implements APIs for recommendation generation and user profile updates.

Components:

  • Recommendation API: Accepts requests from the Django application to generate personalized recommendations based on user profiles and knowledge graphs.

  • Profile Update API: Receives and processes requests to update user profiles with new information collected from social media platforms.

Databases

User Database (Django): A PostgreSQL database that stores user authentication and registration information.

Knowledge graph Database (New4j): Stores RDF-based knowledge graphs representing relationships between users and items of interest.

RDF Representation of Knowledge Graph

Format: Utilizes RDF (Resource Description Framework) to represent the knowledge graph.

Structure: Nodes represent entities (e.g., users, events, places) and edges represent relationships between entities.

Ontology: Defines a set of terms and their relationships, providing a standardized way to represent information.

User Registration Process

  1. When a user registers, the Django application collects basic information and initiates the process.
  2. The Flask application accesses the Neo4j database to retrieve the user’s knowledge graph.
  3. Using a recommendation algorithm, the Flask application generates personalized recommendations based on the user’s profile and knowledge graph.
  4. The recommendations are sent back to the Django application for display to the user.

Recommendation Generation Process

  1. The Django application, upon receiving a user request for recommendations, sends a request to the Flask Recommendation API.
  2. The Flask application accesses the Neo4j database to retrieve the user’s knowledge graph.
  3. Using a recommendation algorithm, the Flask application generates personalized recommendations based on the user’s profile and knowledge graph.
  4. The recommendations are sent back to the Django application for display to the user.

Profile Update Process

  1. When a user updates their profile through the Django interface, the Django application sends a request to the Flask Profile Update API.
  2. The Flask application updates the user’s information in the user database and, if applicable, updates the knowledge graph in the Neo4J database.

The RDF Knowledge Graph is automatically built after a user registers and provides initial information to create the user node in the Neo4j knowledge graph. If the user chooses to use it’s personal Google account, the initial information available is enough to provide the most basic knowledge graph. Any data that the user provides is integrated into the existing knowledge graph, enriching the user’s profile. As the user interacts with the system and provides feedback, the knowledge graph is continuously updated to reflect the user’s evolving preferences and interests. Machine learning algorithms may be employed to improve the accuracy of recommendations based on user feedback. The architecture ensures a separation of concerns, with Django handling user interactions and authentication and Flask managing recommendation generation and profile updates. the Neo4j database serves as the central repository for the RDF knowledge graph, enabling efficient storage and retrieval of complex relationships. The use of RDF allows for a flexible and standardized representation of information, supporting the system’s goal of providing personalized recommendations based on a rich knowledge graph.

Data Model

Use Node

  • Type: schema.org/Person
  • Properties:
    • schema:name - User’s name
    • schema:birthDate - User’s birthday
    • schema:gender - User’s gender
    • schema:address - User’s location
    • schema:jobTitle - User’s job title
    • schema:worksFor - Organization where the user works
    • schema:workLocation: Location of the user’s workplace
    • schema:yearsWorked: Number of years the user has worked

Education Node

  • Type: schema.org/EducationalOrganization
  • Properties:
    • schema:alumniOf - University or college the user attended
    • schema:highschool - High school the user attended

Hobbies and Interests Node

  • Type: schema.org/Person
  • Properties:
    • custom.hobbies - List of hobbies
    • schema.memberOf - Groups that the user is a member of
    • schema:performerIn: Social events attended by the user

Contact Information Node

  • Type: schema.org/Person
  • Properties:
    • schema:telephone - User’s phone number
    • schema:email - User’s email addresses

Media Preferences node

  • Type: schema.org/Person
  • Properties:
    • custom:moviesLiked - List of movies liked by the user
    • custom:tvShowsLiked - List of TV shows liked by the user
    • custom:artistsLiked - List of artists liked by the user
    • custom:musicLiked - List of music genres liked by the user
    • custom:booksLiked - List of books liked by the user
    • custom:sportsLiked - List of sports liked by the user
    • custom:foodLiked - List of food liked by the user
    • custom:applicationsLiked - List of applications liked by the user
    • custom:gamesLiked - List of games liked by the user

Social Media Profiles Node

  • Type: schema.org/Person
  • Properties:
    • custom:socialMediaProfiles - List of social media profiles belonging to the user
    • schema:url - User’s personal website, if any

Family Node

  • Type: schema.org/Person
  • Properties:
    • schema:children - Children of the user
    • schema:spouse - Spouse of the user
    • schema:parents - Parents of the user

Skills Node

  • Type: schema.org/Person
  • Properties:
    • custom:skills - List of skills possessed by the user

This RDF knowledge graph structure provides a flexible and extensible representation of a user’s social media knowledge, capturing a wide range of information and relationships for generating personalized recommendations. Each node corresponds to a specific aspect of the user’s profile, allowing for detailed and context-rich recommendations.
